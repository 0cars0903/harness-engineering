# 글쓰기 파이프라인 템플릿

## 5단계 구체적 행동

| 단계 | 구체적 행동 | 출력물 | 사용 도구/방법론 |
|------|-----------|--------|----------------|
| Input | 독자 분석, 목적 정의, SSOT(사실 정보) 수집 | CPS + 독자 프로필 + SSOT | AskUserQuestion, CPS 프레임 |
| Plan | 역량 분석(자소서), 핵심 메시지 도출, 논증 전략 수립 | 메시지 전략서 | TodoWrite, 3C4P/STAR 선택 |
| Design | 글의 구조(섹션 순서, 분량 배분), 기법 배합 설계 | 글 구조 Blueprint | SKILL.md Blueprint, 3C4P/STAR |
| Execute | Blueprint에 따라 초안 작성 | 글 초안 | writing-harness, job-workspace, docx 스킬 |
| Lint | 문장 품질, 사실 정합성, 구조 준수, 분량 검증 | 첨삭 보고서 + 최종본 | 글쓰기 Lint Rules (W-01~W-08) |

## Input 단계 상세

AskUserQuestion으로 수집할 항목:

1. **Context**: 어떤 매체/채널에 게시되는 글인가? 목적은?
2. **Problem**: 독자가 현재 갖고 있는 인식과 원하는 변화는?
3. **Solution**: 글의 형식(자소서/보고서/블로그 등), 분량 제한, 제출 기한
4. **SSOT 확인**: 사실 정보의 출처 파일을 명시적으로 확인

자소서인 경우 추가 수집:
- 지원 기업과 직무
- JD에서 요구하는 핵심 역량
- 문항 수와 글자 수 제한

## Plan 단계 상세

글의 유형에 따라 전략을 수립한다:

**자소서/에세이:**
1. JD 핵심 키워드 추출 (직무 역량 3~5개)
2. 지원자 경험 중 매칭되는 에피소드 선별
3. 문항별 전략: 3C4P vs STAR 기법 배합 결정
4. 메시지 흐름 설계: 결론 → 맥락 → 행동 → 성과

**보고서/블로그:**
1. 핵심 메시지(테제) 1문장 정의
2. 지지 근거 3~5개 정리
3. 논증 전략: 연역적/귀납적/비교 중 선택
4. 독자의 사전지식 수준에 맞는 설명 깊이 결정

## Design 단계 상세

Blueprint에 포함할 항목:

- **섹션 순서**: 각 섹션의 역할과 배치
- **분량 배분**: 전체 대비 각 섹션의 비율 (예: 도입 10% / 본론 70% / 결론 20%)
- **기법 배합**: 문항별로 적용할 글쓰기 기법 (3C4P, STAR, 두괄식 등)
- **파일명 규칙**: `{기업명}_{문서유형}_{날짜}.docx`
- **톤앤매너**: 격식 수준, 인칭, 어조

## Execute 단계 위임

- 자소서 → `writing-harness` + `job-workspace` 스킬
- 보고서/일반 문서 → `writing-harness` + `docx` 스킬
- 블로그/기고문 → `writing-harness` 스킬

## Lint 규칙

| ID | 규칙 | 검증 방법 |
|----|------|----------|
| W-01 | 문장 길이 ≤ 60자 (국문 기준) | 문장 단위 길이 측정 |
| W-02 | 수동태 비율 ≤ 15% | 수동태 표현 카운트 / 전체 문장 |
| W-03 | 헤딩 계층 최대 H3까지 | 마크다운/문서 구조 확인 |
| W-04 | 한 단락 최대 5문장 | 단락별 문장 수 카운트 |
| W-05 | 금지 어휘 사용 없음 | 금지 어휘 리스트 대조 |
| W-06 | 첫 문장은 핵심 결론 (두괄식) | 각 섹션 첫 문장이 결론인지 확인 |
| W-07 | 숫자는 반드시 단위와 함께 기술 | 숫자 뒤 단위 존재 여부 |
| W-08 | 모든 사실 정보는 SSOT와 1:1 대조 통과 | SSOT 파일 vs 본문 대조 |

### W-01 상세: 문장 길이 제한

국문 기준 60자, 영문 기준 25단어를 초과하는 문장을 검출한다.
초과 시 문장 분리 또는 불필요한 수식어 제거를 제안한다.

### W-05 상세: 금지 어휘

다음 어휘는 사용을 금지한다:
- 추상적 표현: "다양한", "여러 가지", "많은 것들"
- 진부한 표현: "~에 기여하고 싶습니다", "열정을 가지고"
- 검증 불가 표현: "최고의", "가장 ~한", "독보적인" (객관적 근거 없이)

### W-08 상세: SSOT 정합성

본문에 등장하는 모든 기술명, 수치, 프로젝트명, 날짜를 추출하여 SSOT 파일과 1:1 대조한다.
SSOT에 없는 정보는 `[확인 필요]` 태그를 붙여 사용자에게 보고한다.

## 파이프라인 인스턴스 예시

```
📌 업무 유형: 글쓰기
📌 이해관계자: 지원자(나) / 채용담당자 / 팀 리더

━━━ 자동 생성된 파이프라인 ━━━

[Input] AskUserQuestion으로 수집:
  - Context: 어떤 기업, 어떤 직무에 지원하는가?
  - Problem: JD에서 요구하는 핵심 역량은?
  - Solution: 문항 수, 글자 수 제한, 제출 기한
  - SSOT 확인: 지원자_배경_정보.md, project_list.md 로드

[Plan] 역량-경험 매칭 전략:
  1. JD 핵심 키워드 추출 (직무 역량 3~5개)
  2. 지원자 경험 중 매칭 에피소드 선별
  3. 문항별 기법: 3C4P vs STAR 배합 결정
  4. 메시지 흐름: 결론 → 맥락 → 행동 → 성과

[Design] 글 구조 Blueprint:
  - 섹션: 핵심 결론 → 상황 → 행동 → 성과 → 지원동기
  - 분량: 10% / 20% / 40% / 20% / 10%
  - 기법: 문항 1(3C4P) + 문항 2(STAR) + 문항 3(3C4P)
  - 파일명: {기업명}_자기소개서_{날짜}.docx

[Execute] writing-harness → job-workspace → docx 스킬 체인

[Lint] 검증 체크리스트:
  □ W-01: 모든 문장 60자 이하
  □ W-02: 수동태 비율 15% 이하
  □ W-06: 모든 문항 첫 문장이 핵심 결론
  □ W-07: 수치는 단위와 함께
  □ W-08: SSOT 정합성 검증 통과
  □ 글자 수 제한 준수
```
