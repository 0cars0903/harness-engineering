# 리서치 파이프라인 템플릿

## 5단계 구체적 행동

| 단계 | 구체적 행동 | 출력물 | 사용 도구/방법론 |
|------|-----------|--------|----------------|
| Input | 리서치 질문(RQ) 정의, 조사 범위/깊이 수준 합의 | CPS + RQ 목록 + 범위 정의 | AskUserQuestion, CPS 프레임 |
| Plan | 소스 선정 전략, 검색 키워드 설계, 수집 기준 수립 | 리서치 계획서 | TodoWrite |
| Design | 리서치 노트 구조, 종합 보고서 형식, 인용 체계 명세 | 리서치 Blueprint | SKILL.md Blueprint |
| Execute | 정보 수집, 소스별 정리, 종합 및 비교 분석 | 리서치 노트 + 종합 보고서 | research-harness, WebSearch, Notion |
| Lint | 출처 신뢰도, 인용 정확성, RQ 답변 완결성, 편향 검증 | 검증 보고서 + 최종 리서치 | 리서치 Lint Rules (R-01~R-05) |

## Input 단계 상세

AskUserQuestion으로 수집할 항목:

1. **Context**: 이 리서치가 필요한 배경은? 어떤 의사결정/프로젝트를 위한 것인가?
2. **Problem**: 현재 알지 못하는 핵심 정보는 무엇인가?
3. **Solution**: 리서치 결과물 형식 (보고서/요약/데이터셋), 깊이 수준 (개관/심층/전문가급)
4. **Research Questions**: 구체적 RQ 목록 (사용자와 함께 정제)

깊이 수준 가이드:
- **개관 (Survey)**: 주제 전반에 대한 빠른 이해. 소스 3~5개, 1~2시간 분량.
- **심층 (Deep Dive)**: 특정 주제에 대한 상세 분석. 소스 10~15개, 다각도 비교.
- **전문가급 (Expert)**: 학술/산업 수준의 종합. 소스 20개 이상, 메타 분석 포함.

## Plan 단계 상세

리서치 계획서에 포함할 항목:

1. **소스 선정 전략**: 어떤 유형의 소스를 우선 탐색하는가
   - 1차 소스: 원 데이터, 공식 문서, 논문
   - 2차 소스: 분석 보고서, 리뷰 기사, 전문가 인터뷰
   - 3차 소스: 백과사전, 교과서, 종합 가이드
2. **검색 키워드 매트릭스**: RQ별 핵심 키워드와 조합
3. **수집 기준**: 발행일 범위, 신뢰도 기준, 언어
4. **정보 수집 템플릿**: 소스별로 기록할 항목 (출처, 핵심 내용, 신뢰도 평가)

## Design 단계 상세

Blueprint에 포함할 항목:

- **리서치 노트 구조**: RQ별로 소스와 발견사항을 정리하는 형식
- **종합 보고서 구조**: 서론 → RQ별 답변 → 종합 인사이트 → 한계점 → 추가 연구 방향
- **인용 체계**: APA/시카고/간략 인용 중 선택
- **시각화**: 비교표, 타임라인, 관계도 등

## Execute 단계 위임

- `research-harness` 스킬을 주 실행자로 사용
- `WebSearch`로 온라인 소스 수집
- Notion MCP로 기존 노트/문서 참조
- 필요시 `xlsx`로 데이터 정리, `docx`/`pptx`로 보고서 작성

## Lint 규칙

| ID | 규칙 | 검증 방법 |
|----|------|----------|
| R-01 | 모든 사실 주장에 출처(URL/논문/날짜)가 있는가 | 주장-출처 1:1 매핑 |
| R-02 | 출처의 발행일이 요구된 기간 내인가 | 날짜 범위 필터링 |
| R-03 | 모든 RQ에 대한 답변이 포함되어 있는가 | RQ 목록 vs 보고서 섹션 대조 |
| R-04 | 단일 소스에 과도하게 의존하지 않는가 (최소 3개 독립 소스) | 출처별 인용 빈도 분석 |
| R-05 | 상반된 관점이 공정하게 다뤄졌는가 | 관점 다양성 확인 |

### R-01 상세: 출처 명시

모든 사실 주장(통계, 인용, 사실 진술)에는 다음이 동반되어야 한다:
- 출처 이름/기관
- URL 또는 DOI
- 발행일 또는 접근일

### R-02 상세: 발행일 검증

리서치 계획서에서 정의한 기간 범위를 벗어나는 출처를 검출한다.
기술/시장 분석의 경우 일반적으로 최근 2년 이내를 권장하되, 사용자 요구에 따라 조정.

### R-03 상세: RQ 완결성

RQ 목록의 각 질문에 대해:
- 보고서에서 해당 질문을 직접 다루는 섹션이 존재하는가
- 답변이 충분히 구체적인가 (단순 "그렇다/아니다"가 아닌 근거 포함)

### R-04 상세: 소스 다양성

전체 인용에서 하나의 소스가 차지하는 비율이 40%를 초과하면 경고.
최소 3개의 독립적 소스(동일 기관/저자가 아닌)에서 교차 검증.

### R-05 상세: 관점 편향 검증

주요 결론에 대해:
- 반대 의견/대안적 관점이 언급되었는가
- 언급된 경우 공정하게 다뤄졌는가 (strawman이 아닌가)
- 결론이 단일 관점에 치우치지 않았는가

## 파이프라인 인스턴스 예시

```
📌 업무 유형: 리서치
📌 이해관계자: 프로덕트 매니저(의사결정) / 개발팀(기술 선택) / 경영진(투자 판단)

━━━ 자동 생성된 파이프라인 ━━━

[Input] AskUserQuestion으로 수집:
  - Context: 새 프로젝트를 위한 기술 스택 선정 리서치
  - Problem: React vs Vue vs Svelte 중 우리 팀에 최적인 프레임워크?
  - Solution: 심층 수준 비교 보고서, 10페이지 이내
  - RQ 목록:
    1. 각 프레임워크의 성능 벤치마크는?
    2. 우리 팀 규모(5명)에서 학습 곡선은?
    3. 생태계 성숙도와 장기 지원 전망은?
    4. 채용 시장에서의 인력풀 규모는?

[Plan] 리서치 전략:
  1. 소스: 공식 문서, State of JS 2025, GitHub 통계, 벤치마크 사이트
  2. 키워드: "React vs Vue vs Svelte 2025", "frontend framework benchmark"
  3. 기간: 2024~2025 자료 우선
  4. 비교 축: 성능, 학습 곡선, 생태계, 채용

[Design] Blueprint:
  - 구조: 서론 → 개별 분석(3개) → 비교 매트릭스 → 팀 적합성 평가 → 결론
  - 비교표: 5개 축 x 3개 프레임워크 매트릭스
  - 인용: 간략 인용 (저자, 연도, URL)

[Execute] research-harness → WebSearch → xlsx(비교표) → docx 스킬 체인

[Lint] 검증 체크리스트:
  □ R-01: 모든 벤치마크 수치에 출처 명시
  □ R-02: 2024~2025 자료 사용 확인
  □ R-03: 4개 RQ 모두 답변 포함
  □ R-04: 프레임워크당 최소 3개 독립 소스
  □ R-05: 각 프레임워크의 장단점 균형 있게 기술
```
