# Pipeline Designer — HE 메타 파이프라인 설계 스킬

사용자가 문제를 던지면, 실행 전에 **파이프라인을 먼저 설계**하는 메타 스킬이다.
설계와 실행을 분리하여, 사용자가 검토·승인한 후에만 실행에 들어간다.

---

## 핵심 설계 원칙

이 5가지 원칙은 모든 파이프라인 설계에서 절대 위반하지 않는다:

1. **구조는 고정, 내용은 동적** — 5단계(Input → Plan → Design → Execute → Lint)의 순서와 목적은 불변. 각 단계의 구체적 행동만 업무 유형에 따라 달라진다.
2. **설계와 실행의 분리** — 파이프라인 설계(Phase 1~4)가 끝나면 반드시 사용자에게 검토를 요청한다. 승인 없이 Execute 단계로 넘어가지 않는다.
3. **Lint는 하드 가이드** — Lint는 권고가 아닌 강제. 통과하지 못한 결과물은 최종 출력하지 않는다.
4. **Normal Form 수렴** — 같은 파이프라인 인스턴스를 거치면, 어떤 모델이 실행하든 구조적으로 동일한 형태의 결과물이 나와야 한다.
5. **점진적 구체화** — 기초 템플릿이 80%를 커버하고, 나머지 20%는 문제의 특수성에 맞게 동적으로 생성한다.

---

## 메타 디자이너 프로세스: 4단계

사용자가 문제를 던지면, 아래 4단계를 순차적으로 실행하여 파이프라인 인스턴스를 생성한다.

### Phase 1: 문제 분류 (Problem Classification)

사용자의 요청을 분석하여 업무 유형을 판별한다.

| 업무 유형 | 트리거 키워드 | 기본 도메인 |
|-----------|-------------|------------|
| 코딩/빌딩 | 만들어줘, 개발, 사이트, 앱, API, 기능, 배포 | 소프트웨어 엔지니어링 |
| 글쓰기 | 써줘, 자소서, 보고서, 에세이, 블로그, 기고문 | 문서 작성 |
| 기획/분석 | 분석, 기획서, 전략, 비교, 평가, 의사결정 | 비즈니스 분석 |
| 리서치 | 조사, 찾아줘, 트렌드, 벤치마크, 시장조사 | 정보 수집/종합 |
| 디자인/크리에이티브 | 디자인, 슬라이드, 프레젠테이션, 시각화 | 시각 커뮤니케이션 |

**복합 유형 처리**: 하나의 문제가 여러 유형에 걸칠 수 있다. "해커톤 사이트 만들어줘"는 코딩 + 기획의 복합 유형이다. 이 경우 각 유형의 파이프라인을 직렬/병렬로 조합하여 설계한다.

분류 결과를 사용자에게 확인한다: "이 업무를 [유형]으로 분류했습니다. 맞나요?"

### Phase 2: 이해관계자 분석 (Stakeholder Decomposition)

업무 유형에 관계없이, 모든 문제에서 이해관계자를 식별한다. `AskUserQuestion`으로 아래 4가지를 확인한다:

1. **Primary User** — 결과물을 직접 사용하는 사람
2. **Evaluator** — 결과물을 평가/승인하는 사람
3. **Stakeholder** — 결과물에 영향을 받는 간접 이해관계자
4. **Maintainer** — 결과물을 유지보수/관리하는 사람

사용자의 답변이 불충분하면 업무 유형에 맞는 기본값을 제안한다. 예를 들어 자소서라면 Primary User=지원자, Evaluator=채용담당자, Stakeholder=팀 리더, Maintainer=없음(일회성).

### Phase 3: CPS 프레임 자동 생성

이해관계자 분석 결과를 바탕으로 CPS를 구조화한다:

```
[Context] 어떤 상황에서 이 업무가 필요한가?
  - 배경, 도메인, 시간적 제약

[Problem] 지금 어떤 문제/공백이 존재하는가?
  - 이해관계자별 Pain Point
  - 현재 상태 vs 목표 상태의 갭

[Solution] 어떤 형태의 결과물이 필요한가?
  - 결과물 유형 (코드/문서/분석/시각물)
  - 품질 기준 (이해관계자별 성공 조건)
  - 제약 조건 (시간, 분량, 형식, 도구)
```

CPS의 빈칸을 채우기 위해 `AskUserQuestion`으로 부족한 정보를 수집한다. 이미 대화에서 파악된 내용은 건너뛴다.

### Phase 4: 파이프라인 인스턴스 생성

업무 유형 + 이해관계자 + CPS를 결합하여 5단계 각각에 구체적 행동을 채운다.

업무 유형별 기초 템플릿은 `references/` 폴더에 있다:
- 코딩/빌딩 → `references/coding-template.md` 읽기
- 글쓰기 → `references/writing-template.md` 읽기
- 기획/분석 → `references/analysis-template.md` 읽기
- 리서치 → `references/research-template.md` 읽기
- 디자인/크리에이티브 → 코딩 또는 기획 템플릿을 기반으로 동적 생성
- 복합 유형 → 관련 템플릿 모두 읽어 조합

해당 템플릿을 기반으로, 문제의 특수성에 맞게 각 단계의 행동을 커스터마이징한다.

---

## 파이프라인 인스턴스 출력 형식

설계가 완료되면 다음 형식으로 사용자에게 제시한다:

```
📌 업무 유형: [분류 결과]
📌 이해관계자: [Primary] / [Evaluator] / [Stakeholder] / [Maintainer]

━━━ 자동 생성된 파이프라인 ━━━

[Input] 수집 항목:
  - Context: ...
  - Problem: ...
  - Solution: ...
  - 추가 확인 사항: ...

[Plan] 전략:
  1. ...
  2. ...
  3. ...

[Design] Blueprint 명세:
  - 구조: ...
  - 형식: ...
  - 네이밍: ...

[Execute] 실행 방법:
  - 사용할 스킬/도구: ...
  - 실행 순서: ...

[Lint] 검증 체크리스트:
  □ [규칙 ID]: [검증 항목]
  □ ...
```

이 형식을 제시한 후, 반드시 사용자에게 묻는다: "이 파이프라인으로 진행할까요? 수정할 부분이 있으면 말씀해주세요."

---

## 실행 단계 위임

사용자가 파이프라인을 승인하면, Execute 단계에서 적절한 하네스 스킬로 위임한다:

| 업무 유형 | 위임 대상 |
|-----------|----------|
| 코딩/빌딩 | `project-setup` → `api-scaffold` → `tdd-test` 체인 |
| 글쓰기 | `writing-harness` 스킬 |
| 기획/분석 | `analysis-harness` 스킬 |
| 리서치 | `research-harness` 스킬 |
| 디자인 | `pptx` / 시각화 도구 |

위임 시, 파이프라인 인스턴스의 Design 섹션과 Lint 체크리스트를 함께 전달한다.

---

## Lint 단계 실행

Execute가 완료되면, 파이프라인 인스턴스의 Lint 체크리스트를 하나씩 검증한다.
위반 항목이 있으면 자동 교정을 시도하고, 교정 결과를 사용자에게 보고한다.

Lint 규칙의 상세 목록은 각 업무 유형 템플릿의 하단에 정의되어 있다. 업무 유형별 하네스 스킬에도 Lint 규칙이 내장되어 있으므로, 하네스 스킬이 존재하는 유형은 해당 스킬의 Lint를 우선 적용한다.
